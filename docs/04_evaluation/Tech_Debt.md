# Technical Debt Register - IoT 智能仓储监控与告警平台

本文档记录首期验证阶段有意接受的技术债务。登记技术债务的目的不是证明架构有缺陷，而是说明哪些简化是范围控制的结果，以及后续如何偿还。

## 2. 偿还路线图

### 短期：当前验证版本到 v1.1 前

- 为 DEBT-003 补充安全设计说明，明确简化认证与生产级双向认证的差距。
- 为 DEBT-004 在 PoC 中记录缓存容量和补传行为。
- 为 DEBT-005 在压测报告中记录 TimescaleDB 本地写入和查询边界。
- 为 DEBT-008 将告警阈值设计为配置项，而不是写死在说明中。

### 中期：v1.1，0 到 6 个月

- 偿还 DEBT-004：增加 Edge Cache 容量水位、过期清理、补传限速和积压告警。
- 偿还 DEBT-006：完善设备影子 desired/reported 状态模型和版本号。
- 偿还 DEBT-008：实现告警规则版本管理、灰度发布和回滚。
- 偿还 DEBT-010：增加多仓库边缘节点注册和健康状态上报。

### 长期：v2.0，6 到 18 个月

- 偿还 DEBT-001：将服务部署到 Kubernetes 或云容器平台，实现多副本和滚动升级。
- 偿还 DEBT-002：引入生产级 MQTT Broker 高可用方案。
- 偿还 DEBT-003：实现 mTLS、证书轮换和 Topic ACL。
- 偿还 DEBT-007：根据真实吞吐和可靠性目标评估 Kafka、RabbitMQ 或云消息服务。
- 偿还 DEBT-009：实现 OTA 任务编排、批次控制和设备升级状态追踪。

## 3. 债务接受原则

当前阶段接受这些技术债务的原因：

1. 当前阶段重点是架构推理、文档闭环和关键决策验证，不是完整生产系统。
2. 首期验证版本必须保持可完成、可运行，避免范围扩大到完整 IoT 平台。
3. 每项债务都有明确关联 ADR、影响范围和偿还策略。
4. 高优先级债务会在 Phase 4 或 Phase 5 中通过验证证据、演进路线或运行手册进行说明。
