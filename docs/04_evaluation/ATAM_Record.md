# ATAM Record - IoT 智能仓储监控与告警平台

本文档记录对当前边云协同架构的轻量 ATAM 评估。评估目标是识别架构是否能支撑关键质量属性，并为 Phase 4 的 PoC、压测和故障注入提供验证重点。

## 1. 评估范围与目标

评估范围覆盖 Phase 2 已定义的核心架构：

- IoT 设备通过 MQTT 接入边缘 Broker。
- `device-gateway` 负责设备消息校验、边缘缓存和恢复补传。
- `inventory-service` 负责设备影子、库存状态、规则判断和事件发布。
- `alert-service` 负责消费异常事件并生成告警记录。
- TimescaleDB 负责存储传感器时序数据、设备影子、库存状态和告警记录。
- Event Channel 负责异常事件和补货事件传递。

评估目标：

1. 判断当前架构是否覆盖智能仓储场景的关键质量挑战。
2. 找出影响实时性、可靠性、可伸缩性、安全性和可维护性的敏感点。
3. 明确主要权衡点，例如边缘缓存增加可靠性但提升复杂度。
4. 形成风险清单和后续验证重点。

## 2. 业务驱动因素回顾

项目业务背景是物流公司需要对仓库中的传感器、RFID 设备和 AGV 状态设备进行统一接入与管理。系统需要支持库存实时可视化、异常预警、自动补货触发和出入库流程自动化。

关键业务目标：

- 快速发现温度、湿度、重量或库存异常。
- 在设备或网络不稳定时避免关键数据丢失。
- 支持未来多仓库、多设备类型扩展。
- 用可运行验证环境检验关键架构行为，而不是一次性实现完整仓储系统。

关键质量目标来自 `docs/01_adf/Quality_Scenarios.md`：

| QAS | 质量目标 | 评估关注点 |
| --- | --- | --- |
| QAS-001 | 异常温度 5 秒内告警 | 告警链路延迟、事件积压 |
| QAS-002 | 设备消息高吞吐接入 | MQTT Broker、gateway 消费能力 |
| QAS-003 | 云端短暂不可用后的边缘补传 | Edge Cache、补传幂等 |
| QAS-004 | 设备离线后仍可查询最后状态 | 设备影子更新与离线判定 |
| QAS-005 | 传感器历史数据按时间查询 | TimescaleDB 写入和查询模型 |
| QAS-006 | 伪造设备接入被拒绝 | 设备身份认证和 Topic 权限 |
| QAS-007 | 告警规则变更不影响设备接入 | 规则服务边界和事件解耦 |
| QAS-009 | 库存低于阈值触发补货事件 | 库存规则和事件发布 |

## 3. 架构描述

架构描述参考以下 Phase 2 文档：

- `docs/02_views/Context_View.md`
- `docs/02_views/Container_View.md`
- `docs/02_views/Component_View.md`
- `docs/02_views/Dynamic_Views.md`
- `docs/02_views/Deployment_View.md`

当前架构采用边云协同：

```text
IoT 设备
  -> MQTT Broker
  -> device-gateway
  -> inventory-service
  -> Event Channel
  -> alert-service
  -> TimescaleDB
```

关键设计决策来自以下 ADR：

| ADR | 决策 | 架构作用 |
| --- | --- | --- |
| ADR-001 | 选择 MQTT | 支撑设备异步上报和高吞吐接入 |
| ADR-002 | 采用边云协同 | 平衡现场可靠性和云端治理 |
| ADR-003 | 选择 TimescaleDB | 支撑传感器时序数据存储和查询 |
| ADR-004 | 采用设备影子 | 支撑离线设备最后状态查询 |
| ADR-005 | 采用事件驱动告警 | 解耦数据处理和告警生成 |
| ADR-006 | 采用边缘缓存补传 | 提升云端短暂不可用时的数据可靠性 |

## 5. 架构方法分析

### MQTT 接入

ADR-001 使用 MQTT 和 Broker 处理设备消息。该方法适合大量设备异步上报，但 Broker 性能、Topic 规划、认证和 ACL 会直接影响 QAS-002 与 QAS-006。Phase 4 需要用模拟设备压测 Broker 和 `device-gateway` 的消费能力。

### 边云协同

ADR-002 将现场接入、缓存和补传放在边缘侧，将设备影子、库存、告警和长期存储放在云端。该方法提升 QAS-003 的可靠性，但引入边缘缓存容量、补传顺序、幂等键和运维复杂度。

### TimescaleDB

ADR-003 使用 TimescaleDB 统一承载时序数据和普通业务数据。该方法降低原型复杂度，有利于 QAS-005，但在高写入压力下需要批量写入、索引、保留策略和压缩策略。

### 设备影子

ADR-004 使用设备影子保存最后状态和期望状态。该方法直接支撑 QAS-004 和 QAS-008，但需要处理状态版本、新旧消息、离线判定和期望状态确认。

### 事件驱动告警

ADR-005 使用事件通道解耦 `inventory-service` 和 `alert-service`。该方法支撑 QAS-001、QAS-007 和 QAS-009，但事件积压、重复消费和失败重试会影响告警延迟。

### 边缘缓存补传

ADR-006 使用 Edge Cache 和 Replay Worker。该方法支撑 QAS-003，但缓存容量、补传限速和幂等处理是高风险点。

| --- | --- | --- | --- |
| SP-001 | MQTT Broker 最大连接数和消息吞吐 | 可伸缩性、实时性 | MQTT Broker |
| SP-002 | `device-gateway` 消费和转发速率 | 可伸缩性、实时性 | device-gateway |
| SP-003 | Edge Cache 容量和写入速度 | 可靠性 | Edge Cache |

| SP-006 | TimescaleDB 写入和查询索引设计 | 可观测性、性能 | TimescaleDB |
| SP-007 | 设备影子新旧消息判断 | 可靠性、一致性 | Device Shadow Manager |
| SP-008 | 设备认证和 Topic ACL 强度 | 安全性 | MQTT Broker, Device Authenticator |

## 7. 权衡点清单

| ID | 描述 | 正向影响 | 负向影响 |
| --- | --- | --- | --- |
| TP-001 | 选择边云协同而不是全云端 | 提升现场可靠性和离线容忍 | 增加边缘节点运维复杂度 |
| TP-002 | 使用事件驱动告警 | 解耦服务，利于扩展 | 引入事件积压、重复消费和追踪复杂度 |
| TP-003 | 使用 TimescaleDB 统一存储 | 降低首期实现复杂度，便于查询 | 高吞吐场景下需要额外调优 |
| TP-004 | 使用设备影子 | 离线状态可查，支持未来 OTA | 需要维护状态版本和离线判定 |
| TP-005 | 边缘缓存补传 | 减少云端不可用导致的数据丢失 | 需要处理缓存容量和补传风暴 |
| TP-006 | 首期安全策略简化 | 降低首期实现成本 | 生产安全性验证不足 |

## 8. 风险与非风险

### 风险

| 风险ID | 描述 | 影响的质量属性 | 缓解方向 |
| --- | --- | --- | --- |
| RISK-001 | MQTT Broker 或 gateway 吞吐不足 | 可伸缩性、实时性 | 压测并说明本地测试边界 |
| RISK-002 | 告警事件积压导致超过 5 秒 | 实时性 | 监控事件积压，验证异常链路 |
| RISK-003 | 边缘缓存补传产生重复处理 | 可靠性、一致性 | 引入幂等键 |
| RISK-004 | TimescaleDB 单实例无法代表生产规模 | 可伸缩性 | 报告中声明原型边界 |
| RISK-005 | 简化认证无法证明双向认证要求 | 安全性 | 文档设计生产级认证策略 |

### 非风险

| 非风险ID | 描述 | 理由 |
| --- | --- | --- |
| NR-001 | 不实现复杂前端 | 当前阶段关注架构风险、接口契约和核心业务链路，复杂前端不是主要风险 |
| NR-002 | 不接入真实硬件 | 当前验证阶段使用模拟设备检验接入、缓存和告警链路 |
| NR-003 | 不实现完整 WMS/ERP | 系统边界已明确外部系统只作为未来集成对象 |

## 9. 评估结论

当前边云协同架构能够覆盖智能仓储系统的核心架构决策点，并能与 Phase 1 的质量属性场景形成追溯关系。主要风险集中在四类：高吞吐设备接入、5 秒告警链路、边缘缓存补传可靠性、以及安全策略简化。

Phase 4 应优先验证：

1. QAS-001：异常温度上报到告警生成是否能在 5 秒内完成。
2. QAS-003：云端不可用时 Edge Cache 是否能缓存并恢复补传。
3. QAS-002：本地验证环境在可控规模下的 MQTT 接入吞吐表现。
4. QAS-006：设备身份校验的简化实现和生产级安全设计之间的差距。
