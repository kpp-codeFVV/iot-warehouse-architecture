# ADR-003：选择 TimescaleDB 存储传感器时序数据

## 状态

Accepted

## 背景

仓储传感器会按时间持续上报温度、湿度、重量和设备状态。这类数据需要按设备、仓库和时间窗口查询，也需要支持后续压测、告警分析和运行证据收集。系统需要选择适合课程原型且能解释时序数据需求的存储方案。

## 决策驱动因素

- QAS-005：传感器历史数据按时间查询。
- 约束：本地 Docker 原型需要部署简单，团队需要能快速理解和查询数据。

## 可选方案

### 方案A：InfluxDB

- 描述：专门面向时序数据的数据库。
- 优点：时序能力成熟，查询和指标分析能力强。
- 缺点：与普通业务数据需要分库存储，课程原型中需要额外学习查询语法和数据模型。
- 对质量属性的影响：可观测性强，但增加维护和学习成本。

### 方案B：TimescaleDB

- 描述：PostgreSQL 扩展，以 hypertable 支持时序数据。
- 优点：兼容 PostgreSQL，既能存时序数据，也能存设备影子、库存和告警等普通业务数据。
- 缺点：极端时序规模下需要更细致的分区和压缩策略。
- 对质量属性的影响：兼顾可维护性、可观测性和原型可部署性。

### 方案C：TDengine

- 描述：面向物联网和工业场景的时序数据库。
- 优点：IoT 场景针对性强，适合大规模设备数据。
- 缺点：课程原型中团队学习成本较高，生态和通用 SQL 经验不如 PostgreSQL 普遍。
- 对质量属性的影响：可伸缩性潜力强，但原型风险较高。

## 决策结果

选择方案B：TimescaleDB。原型中使用单个 TimescaleDB 容器保存遥测数据、设备影子、库存状态和告警记录。正式架构中说明生产环境可按仓库、设备类型和时间窗口做分区、压缩和保留策略。

## 后果

### 正面影响

- 降低课程原型部署和学习成本。
- 支持时间范围查询和聚合分析。
- 业务数据和时序数据可在同一 PostgreSQL 生态下管理。

### 负面影响/权衡

- 高吞吐写入下需要优化批量写入、索引和保留策略。
- 单实例原型不能代表真实生产容量。

### 引入的技术债务

原型阶段不实现复杂数据保留、压缩和分片策略，需要在演进路线中补充。

## 验证方式

- 原型中写入传感器遥测数据并按时间窗口查询。
- 在 `LoadTest_Report.md` 中记录本地写入和查询表现。
- 在 `QA_Traceability.md` 中将 QAS-005 关联到 TimescaleDB 查询证据。
