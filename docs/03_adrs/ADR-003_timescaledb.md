# ADR-003：选择 TimescaleDB 存储传感器时序数据

## 状态

Accepted

## 背景

仓储传感器会按时间持续上报温度、湿度、重量和设备状态。这类数据后续需要按设备、仓库和时间窗口查询，也会用于压测、告警分析和运行证据收集。因此需要明确时序数据的目标存储方案，同时说明当前原型的实现边界。

## 决策驱动因素

- QAS-005：传感器历史数据按时间查询。
- 约束：本地 Docker 环境需要部署简单，团队需要能快速理解数据模型和查询方式。

## 可选方案

### 方案A：InfluxDB

- 描述：专门面向时序数据的数据库。
- 优点：时序能力成熟，查询和指标分析能力强。
- 缺点：与普通业务数据需要分库存储，团队需要额外学习查询语法和数据模型。
- 对质量属性的影响：可观测性强，但增加维护和学习成本。

### 方案B：TimescaleDB

- 描述：PostgreSQL 扩展，以 hypertable 支持时序数据。
- 优点：兼容 PostgreSQL，既能存时序数据，也能存设备影子、库存和告警等普通业务数据。
- 缺点：极端时序规模下需要更细致的分区和压缩策略。
- 对质量属性的影响：兼顾可维护性、可观测性和原型可部署性。

### 方案C：TDengine

- 描述：面向物联网和工业场景的时序数据库。
- 优点：IoT 场景针对性强，适合大规模设备数据。
- 缺点：团队学习成本较高，生态和通用 SQL 经验不如 PostgreSQL 普遍。
- 对质量属性的影响：可伸缩性潜力强，但原型风险较高。

## 决策结果

选择方案B：TimescaleDB。Docker Compose 环境中保留 TimescaleDB 容器，作为时序数据和后续生产化存储的目标方案。当前 FastAPI 原型为了快速演示主链路，仍使用轻量 JSON/JSONL 文件保存运行数据；并发压测已经暴露出这种简化不适合高并发写入。正式架构中应将遥测、库存、告警等数据迁入 TimescaleDB/PostgreSQL，并按仓库、设备类型和时间窗口设计分区、压缩和保留策略。

## 后果

### 正面影响

- 与 Docker Compose 环境兼容，便于后续从轻量文件存储迁移到数据库。
- 支持时间范围查询和聚合分析。
- 业务数据和时序数据可在同一 PostgreSQL 生态下管理。

### 负面影响/权衡

- 高吞吐写入下需要优化批量写入、索引和保留策略。
- 当前 JSON/JSONL 存储只能用于演示，不能承受并发写入。
- 单实例数据库也不能代表真实生产容量。

### 引入的技术债务

当前阶段接受两项债务：一是原型仍使用轻量文件存储，二是 TimescaleDB 未实现复杂数据保留、压缩和分片策略。这两项都需要在演进路线中补充。

## 验证方式

- 在原型中记录传感器遥测、库存和告警数据，验证主链路可运行。
- 在 `LoadTest_Report.md` 中记录本地写入表现和 JSON 文件并发写入瓶颈。
- 在 `QA_Traceability.md` 中将 QAS-005 关联到存储选型、部署视图和 LoadTest 边界说明。
