# ADR-003：选择 TimescaleDB 存储传感器时序数据

## 状态

Accepted

## 背景

仓储传感器会按时间持续上报温度、湿度、重量和设备状态。这类数据后续需要按设备、仓库和时间窗口查询，也会用于压测、告警分析和运行证据收集。因此需要明确时序数据的目标存储方案，同时说明当前原型的实现边界。

## 决策驱动因素

- QAS-005：传感器历史数据按时间查询。
- 约束：Docker 环境需要部署简单，团队需要能快速理解数据模型和查询方式。

## 可选方案

### 方案A：InfluxDB

- 描述：专门面向时序数据的数据库。
- 优点：时序能力成熟，查询和指标分析能力强。
- 缺点：与普通业务数据需要分库存储，团队需要额外学习查询语法和数据模型。
- 对质量属性的影响：可观测性强，但增加维护和学习成本。

### 方案B：TimescaleDB

- 描述：PostgreSQL 扩展，以 hypertable 支持时序数据。
- 优点：兼容 PostgreSQL，既能存时序数据，也能存设备影子、库存和告警等普通业务数据。
- 缺点：极端时序规模下需要更细致的分区和压缩策略。
- 对质量属性的影响：兼顾可维护性、可观测性和原型可部署性。

### 方案C：TDengine

- 描述：面向物联网和工业场景的时序数据库。
- 优点：IoT 场景针对性强，适合大规模设备数据。
- 缺点：团队学习成本较高，生态和通用 SQL 经验不如 PostgreSQL 普遍。
- 对质量属性的影响：可伸缩性潜力强，但原型风险较高。

## 决策结果

选择方案B：TimescaleDB。系统使用 TimescaleDB/PostgreSQL 存储设备消息、设备影子、库存状态、异常事件和告警记录。这样可以在同一 PostgreSQL 生态下同时处理时序数据和普通业务数据，降低本地部署复杂度，并为后续时间窗口查询、压缩、保留策略和备份恢复留下演进空间。

## 后果

### 正面影响

- 与 Docker Compose 环境兼容，便于本地验证和后续生产化部署。
- 支持时间范围查询和聚合分析。
- 业务数据和时序数据可在同一 PostgreSQL 生态下管理。

### 负面影响/权衡

- 高吞吐写入下需要优化批量写入、索引和保留策略。
- 单实例数据库不能代表真实生产容量。
- 高吞吐目标下仍需要批量写入、连接池调优和消息队列削峰。

### 引入的技术债务

当前阶段接受的主要数据债务是 TimescaleDB 仍为单节点部署，尚未实现 hypertable 分区、压缩、保留策略、备份恢复演练和生产级容量规划。

## 验证方式

- 在本地验证环境中记录设备消息、设备影子、库存、事件和告警数据，验证主链路可运行。
- 在 `LoadTest_Report.md` 中记录 10,000 设备接入和 TimescaleDB/PostgreSQL 并发写入表现。
- 在 `QA_Traceability.md` 中将 QAS-005 关联到存储选型、部署视图和 LoadTest 边界说明。
