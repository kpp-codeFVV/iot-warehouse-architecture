# ADR-002：采用边云协同架构划分 Edge 与 Cloud 职责

## 状态

Accepted

## 背景

智能仓储场景中，设备分布在仓库现场，网络可能出现抖动或短暂中断。如果所有设备数据都直接依赖云端处理，云端不可用时现场数据容易丢失，异常响应也可能变慢。但如果全部逻辑都放在边缘侧，统一治理、长期存储和跨仓库分析会变复杂。因此需要明确 Edge 与 Cloud 的职责边界。

## 决策驱动因素

- QAS-002：设备消息高吞吐接入。
- QAS-003：云端短暂不可用后的边缘补传。
- QAS-008：批量设备升级不影响在线采集。
- 约束：原型需要控制复杂度，同时覆盖题目要求的 Edge 与云端职责划分。

## 可选方案

### 方案A：全云端集中式

- 描述：设备消息直接进入云端 Broker 和业务服务。
- 优点：架构简单，集中管理方便。
- 缺点：现场网络异常时可靠性较弱，边缘计算价值体现不足。
- 对质量属性的影响：可维护性较好，但可靠性和低延迟能力不足。

### 方案B：边云协同

- 描述：边缘侧负责设备接入、基础校验、短期缓存和补传；云端负责设备影子、库存状态、告警记录、长期存储和统一 API。
- 优点：兼顾现场可靠性和云端治理能力。
- 缺点：需要维护边缘侧和云端之间的职责边界。
- 对质量属性的影响：提升可靠性、实时性和可伸缩性。

### 方案C：全边缘自治

- 描述：仓库现场边缘节点完成绝大多数业务逻辑。
- 优点：现场自治能力强，对云端依赖低。
- 缺点：跨仓库统一管理和数据分析困难，部署维护成本高。
- 对质量属性的影响：提升局部可用性，但降低集中可维护性。

## 决策结果

选择方案B：边云协同。边缘层部署 MQTT Broker、`device-gateway` 和 Edge Cache；云端部署 `inventory-service`、`alert-service`、Event Channel 和 TimescaleDB。

## 后果

### 正面影响

- 云端短暂不可用时，边缘侧仍可接收设备消息并缓存。
- 云端保留统一数据治理和查询能力。
- 便于在部署视图中说明仓库现场网络和云端服务网络。

### 负面影响/权衡

- 边缘侧需要额外管理缓存、补传和健康检查。
- 需要定义清晰的数据幂等策略，避免补传造成重复处理。

### 引入的技术债务

原型阶段边缘节点只模拟单仓库部署，后续需要补充多仓库边缘节点注册、配置下发和监控接入。

## 验证方式

- 在 PoC 中模拟云端服务不可用，验证 `device-gateway` 将消息写入 Edge Cache。
- 云端恢复后验证消息能够按顺序补传。
- 在部署视图中说明生产拓扑和 Docker Compose 原型映射。
