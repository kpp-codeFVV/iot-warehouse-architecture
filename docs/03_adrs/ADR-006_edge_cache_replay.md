# ADR-006：采用边缘缓存与恢复补传机制

## 状态

Accepted

## 背景

仓库现场设备仍可能在云端服务不可用时持续产生数据。如果这些数据直接丢弃，会影响库存状态、告警判断和历史分析。设备端能力有限，不能假设每类设备都具备可靠缓存和补传能力。因此需要在边缘网关侧设计统一缓存与恢复补传机制。

## 决策驱动因素

- QAS-003：云端短暂不可用后的边缘补传。
- 约束：原型需要通过 PoC 展示可靠性，而不是只在文档中声明。

## 可选方案

### 方案A：云端不可用时丢弃消息

- 描述：转发失败的设备消息直接丢弃。
- 优点：实现最简单。
- 缺点：关键遥测数据丢失，可靠性无法满足题目要求。
- 对质量属性的影响：严重削弱可靠性。

### 方案B：只依赖设备端缓存

- 描述：设备自行缓存未成功上报的数据。
- 优点：边缘网关实现简单。
- 缺点：不同设备能力不一致，RFID 或低功耗传感器未必支持可靠缓存。
- 对质量属性的影响：可靠性不可控。

### 方案C：边缘网关缓存并补传

- 描述：`device-gateway` 在云端不可用时写入 Edge Cache，恢复后按顺序补传。
- 优点：统一管理缓存和补传，便于观测和 PoC 验证。
- 缺点：边缘节点需要管理缓存容量、过期策略和幂等键。
- 对质量属性的影响：提升可靠性和数据完整性。

## 决策结果

选择方案C：边缘网关缓存并补传。`device-gateway` 为每条消息生成幂等键，转发失败时写入 Edge Cache。云端健康检查恢复后，Replay Worker 按原始时间顺序补传消息。

## 后果

### 正面影响

- 云端短暂不可用时不丢关键消息。
- 可通过 PoC 直接验证可靠性场景。
- 缓存积压量可以成为运维观测指标。

### 负面影响/权衡

- 边缘缓存可能占用磁盘，需要限制容量和保留时间。
- 补传可能造成短时间流量突增，需要限速。

### 引入的技术债务

原型阶段缓存策略较简单，生产环境需要补充容量水位、过期清理、补传限速和告警。

## 验证方式

- 停止 `inventory-service` 后继续发送设备消息，验证消息进入 Edge Cache。
- 恢复 `inventory-service` 后验证 Replay Worker 补传成功。
- 在 `PoC_Report.md` 中记录实验步骤、结果和对 ADR 的影响。
