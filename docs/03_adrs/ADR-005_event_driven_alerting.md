# ADR-005：采用事件驱动告警流水线

## 状态

Accepted

## 背景

异常告警是本项目最重要的核心业务链路。设备数据进入系统后，需要快速判断异常并生成告警。如果告警逻辑和设备接入逻辑强耦合，后续新增通知、报表、审计等消费者时会频繁修改接入服务。系统需要一种能支撑实时性和可修改性的告警流水线。

## 决策驱动因素

- QAS-001：异常温度 5 秒内告警。
- QAS-007：告警规则变更不影响设备接入。
- QAS-009：库存低于阈值触发补货事件。
- 约束：原型需要清楚展示同步通信和异步通信两种服务间通信方式。

## 可选方案

### 方案A：同步调用告警服务

- 描述：`inventory-service` 发现异常后直接 HTTP 调用 `alert-service`。
- 优点：实现简单，调用链直观。
- 缺点：告警服务故障会影响数据处理链路，服务耦合强。
- 对质量属性的影响：降低可用性和可修改性。

### 方案B：数据库轮询

- 描述：`inventory-service` 写入异常记录，`alert-service` 定时轮询数据库。
- 优点：实现成本低，不需要消息通道。
- 缺点：实时性受轮询间隔影响，数据库负担增加。
- 对质量属性的影响：不利于 5 秒告警目标。

### 方案C：事件驱动

- 描述：`inventory-service` 发布异常事件，`alert-service` 异步消费并生成告警。
- 优点：生产者和消费者解耦，支持扩展通知、报表和审计消费者。
- 缺点：需要维护事件 Schema、消费位点和失败重试策略。
- 对质量属性的影响：提升实时性、可伸缩性和可修改性。

## 决策结果

选择方案C：事件驱动。`inventory-service` 将 `HIGH_TEMPERATURE`、`REPLENISHMENT_REQUIRED` 等事件发布到 Event Channel，`alert-service` 消费异常事件并生成告警记录。

## 后果

### 正面影响

- 告警服务故障不直接阻塞设备接入。
- 后续增加通知服务或报表服务时不需要修改设备接入链路。
- 事件/消息契约清晰，异步通信链路便于扩展和追踪。

### 负面影响/权衡

- 需要定义事件 JSON Schema。
- 需要处理重复消费、消费失败和事件积压。

### 引入的技术债务

当前验证环境使用 TimescaleDB/PostgreSQL 事件表承载事件通道，生产环境需要根据吞吐目标评估 Kafka、RabbitMQ 或云消息服务。

## 验证方式

- 验证流程中发送异常温度数据，确认 5 秒内生成告警。
- 通过事件 Schema 和动态视图说明告警事件结构和处理链路。
- 在 `FaultInjection.md` 中模拟告警服务短暂不可用，观察事件积压和恢复消费。
