# ADR-004：采用设备影子模式管理设备状态

## 状态

Accepted

## 背景

IoT 设备可能因为网络、电量或现场环境问题暂时离线。如果系统只依赖实时上报数据，设备离线后管理员无法知道设备最后状态，也不利于未来 OTA 和配置下发。系统需要一种方式在云端保存设备状态副本。

## 决策驱动因素

- QAS-004：设备离线后仍可查询最后状态。
- QAS-008：批量设备升级不影响在线采集。
- 约束：原型需要能直观展示设备状态，而不是只展示原始遥测数据。

## 可选方案

### 方案A：只保存实时上报数据

- 描述：每次设备上报后只记录遥测表，不维护设备状态副本。
- 优点：实现简单。
- 缺点：离线状态查询和期望配置管理困难。
- 对质量属性的影响：可靠性和可维护性不足。

### 方案B：设备影子

- 描述：云端维护每个设备的最后已知状态、在线状态、最后上报时间和期望状态。
- 优点：离线后仍可查询状态，支持未来 OTA 和配置管理。
- 缺点：需要处理状态新旧、幂等更新和离线判定。
- 对质量属性的影响：提升可靠性和可维护性。

### 方案C：每次直接查询设备

- 描述：管理员查询状态时直接向设备发起请求。
- 优点：理论上状态最新。
- 缺点：设备可能离线，网络延迟高，不适合大量设备。
- 对质量属性的影响：实时查询可靠性差，可伸缩性弱。

## 决策结果

选择方案B：设备影子。`inventory-service` 维护设备影子，保存 reported state、desired state、lastSeenAt 和 connectionStatus。

## 后果

### 正面影响

- 设备离线后仍可查询最后状态。
- 支持未来 OTA、配置下发和批量设备管理。
- 库存和告警逻辑可以基于稳定的设备状态模型。

### 负面影响/权衡

- 需要定义设备状态更新的幂等规则。
- 需要处理设备长时间不上报时的离线标记。

### 引入的技术债务

原型阶段只实现简化设备影子，生产环境需要补充 desired/reported 状态版本号和配置确认机制。

## 验证方式

- 原型中模拟设备上报后查询设备影子。
- 停止某设备上报后，验证 API 仍能返回最后一次已知状态。
- 在动态视图和 QA 追溯矩阵中关联 QAS-004。
