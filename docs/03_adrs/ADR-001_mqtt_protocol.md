# ADR-001：选择 MQTT 作为 IoT 设备接入协议

## 状态

Accepted

## 背景

系统需要接入大量仓库 IoT 设备，包括温湿度传感器、重量传感器、RFID 扫码设备和 AGV 状态设备。这些设备会持续上报小体积遥测消息，消息频率高，设备能力差异大，网络质量也可能不稳定。接入协议需要轻量、成熟，并支持大量设备向平台发布消息。

## 决策驱动因素

- QAS-002：设备消息高吞吐接入。
- QAS-006：伪造设备接入被拒绝。
- 约束：本地原型需要容易部署，课程答辩中需要能清楚解释协议选择。

## 可选方案

### 方案A：MQTT

- 描述：面向物联网场景的发布/订阅协议，通过 Broker 转发消息。
- 优点：轻量、成熟、生态丰富，适合设备持续上报；Mosquitto 可快速搭建本地原型。
- 缺点：需要部署 Broker；安全策略需要配合认证、ACL 和 TLS 设计。
- 对质量属性的影响：提升可伸缩性和实时性，有利于设备高吞吐接入。

### 方案B：CoAP

- 描述：基于 UDP 的轻量请求/响应协议。
- 优点：协议开销低，适合受限设备。
- 缺点：发布/订阅和 Broker 生态不如 MQTT 直观；课程原型实现复杂度更高。
- 对质量属性的影响：有利于轻量通信，但对高吞吐消息汇聚和演示可解释性帮助较弱。

### 方案C：AMQP

- 描述：面向消息中间件的企业级消息协议。
- 优点：可靠消息语义强，适合企业系统集成。
- 缺点：协议和客户端相对重，对低功耗或受限 IoT 设备不够友好。
- 对质量属性的影响：提升消息可靠性，但增加设备端复杂度。

## 决策结果

选择方案A：MQTT。设备通过 MQTT Topic 发布遥测数据，边缘侧使用 Mosquitto Broker 接收和分发消息，`device-gateway` 订阅设备 Topic 并完成校验、缓存和转发。

## 后果

### 正面影响

- 符合 IoT 设备持续上报场景。
- 支持发布/订阅模式，设备和业务服务解耦。
- 本地原型容易通过 Mosquitto 和 Python MQTT 客户端实现。

### 负面影响/权衡

- Broker 成为关键基础设施，需要设计重启、监控和权限控制。
- 安全性不能只依赖协议本身，需要配套设备凭证、Topic ACL 和 TLS。

### 引入的技术债务

课程原型阶段只做简化认证，生产环境需要完善双向 TLS、证书轮换和设备级 Topic 权限。

## 验证方式

- 在原型中使用模拟设备向 MQTT Broker 发布遥测消息。
- 验证 `device-gateway` 能订阅并处理消息。
- 在 `LoadTest_Report.md` 中记录本地模拟设备吞吐结果和适用边界。
