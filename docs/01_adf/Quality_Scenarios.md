## QAS-001：异常温度 5 秒内告警

| 要素 | 描述 |
| --- | --- |
| 刺激源 | 温湿度传感器 |
| 刺激 | 设备上报温度超过仓库安全阈值，例如 38 摄氏度 |
| 环境 | 系统处于正常运行状态，MQTT Broker、device-gateway、inventory-service 和 alert-service 均可用 |
| 制品 | 设备接入链路、设备影子、异常事件通道、告警服务 |
| 响应 | 系统接收遥测数据，更新设备影子，发布异常事件，并生成 HIGH_TEMPERATURE 告警 |
| 响应度量 | 从设备消息时间戳到告警记录创建时间不超过 5 秒 |

**关联 ADR**：ADR-005  
**优先级**：Must Have

## QAS-002：设备消息高吞吐接入

| 要素 | 描述 |
| --- | --- |
| 刺激源 | 大量仓库 IoT 设备 |
| 刺激 | 多个仓库中的传感器持续上报温湿度、重量、RFID 和 AGV 状态消息 |
| 环境 | 开学季或促销季仓储高峰，设备接入量接近设计峰值 |
| 制品 | MQTT Broker、device-gateway、事件通道 |
| 响应 | Broker 接收发布消息，device-gateway 按 Topic 消费并转发有效数据，系统通过水平扩展网关实例提升处理能力 |
| 响应度量 | 架构设计目标支持 10,000+ 设备接入和 50,000 msg/s；本地压测报告记录可达到的测试吞吐和适用边界 |

**关联 ADR**：ADR-001, ADR-002  
**优先级**：Must Have

## QAS-003：云端短暂不可用后的边缘补传

| 要素 | 描述 |
| --- | --- |
| 刺激源 | 云端服务故障或网络抖动 |
| 刺激 | device-gateway 无法将设备消息转发到 inventory-service |
| 环境 | 仓库边缘侧仍能接收设备消息，云端服务短暂不可用 |
| 制品 | device-gateway、边缘缓存、补传任务 |
| 响应 | 边缘网关将消息写入本地缓存，云端恢复后按原始时间顺序补传 |
| 响应度量 | 云端恢复后 60 秒内补传缓存中的关键消息，补传过程中消息不重复处理 |

**关联 ADR**：ADR-002, ADR-006  
**优先级**：Must Have

## QAS-004：设备离线后仍可查询最后状态

| 要素 | 描述 |
| --- | --- |
| 刺激源 | IoT 设备 |
| 刺激 | 某设备停止上报数据超过离线阈值 |
| 环境 | 设备网络不稳定，云端服务正常运行 |
| 制品 | 设备影子、设备状态 API |
| 响应 | 系统将设备标记为 offline，并保留最后一次已知状态、最后上报时间和期望配置 |
| 响应度量 | 设备离线后，API 仍能在 1 秒内返回该设备最后一次已知状态 |

**关联 ADR**：ADR-004  
**优先级**：Must Have

## QAS-005：传感器历史数据按时间查询

| 要素 | 描述 |
| --- | --- |
| 刺激源 | 运维人员 |
| 刺激 | 查询某仓库过去 1 小时的温度变化 |
| 环境 | 系统积累了连续传感器遥测数据 |
| 制品 | TimescaleDB、遥测数据表、查询 API |
| 响应 | 系统按设备、仓库和时间范围过滤数据，并返回时间序列结果 |
| 响应度量 | 本地原型环境下，查询 1 小时窗口内的单设备数据响应时间不超过 2 秒 |

**关联 ADR**：ADR-003  
**优先级**：Should Have

## QAS-006：伪造设备接入被拒绝

| 要素 | 描述 |
| --- | --- |
| 刺激源 | 未授权客户端 |
| 刺激 | 客户端尝试使用不存在的设备 ID 或错误凭证发布 MQTT 消息 |
| 环境 | 系统开启设备身份校验策略 |
| 制品 | MQTT Broker、device-gateway、设备注册表 |
| 响应 | 系统拒绝未授权消息，记录安全事件，不更新设备影子 |
| 响应度量 | 未授权消息拒绝率为 100%，安全事件可在日志或 API 中查询 |

**关联 ADR**：ADR-001  
**优先级**：Should Have

## QAS-007：告警规则变更不影响设备接入

| 要素 | 描述 |
| --- | --- |
| 刺激源 | 开发人员或运维人员 |
| 刺激 | 将高温告警阈值从 38 摄氏度调整为 35 摄氏度 |
| 环境 | 设备持续上报数据，接入链路不能停机 |
| 制品 | 告警规则、inventory-service、alert-service |
| 响应 | 系统更新告警规则后，新阈值用于后续异常判断，device-gateway 继续接收设备消息 |
| 响应度量 | 规则变更过程中 MQTT 消息接入不中断，后续异常判断使用新阈值 |

**关联 ADR**：ADR-005  
**优先级**：Should Have

## QAS-008：批量设备升级不影响在线采集

| 要素 | 描述 |
| --- | --- |
| 刺激源 | 运维人员 |
| 刺激 | 对同一仓库的一批设备下发 OTA 升级任务 |
| 环境 | 仓库仍有在线设备持续上报数据 |
| 制品 | 设备影子、边缘网关、OTA 策略 |
| 响应 | 系统将 OTA 目标版本写入设备期望状态，并按批次控制升级，不阻塞其他设备数据采集 |
| 响应度量 | OTA 任务设计为批量分组执行，未参与当前批次的在线设备继续正常上报 |

**关联 ADR**：ADR-004, ADR-002  
**优先级**：Should Have

## QAS-009：库存低于阈值触发补货事件

| 要素 | 描述 |
| --- | --- |
| 刺激源 | 重量传感器或 RFID 扫码设备 |
| 刺激 | 某货架重量或库存计数低于补货阈值 |
| 环境 | 系统正常运行，库存规则已配置 |
| 制品 | inventory-service、库存状态、补货事件 |
| 响应 | 系统更新库存状态，并生成 REPLENISHMENT_REQUIRED 事件 |
| 响应度量 | 库存低于阈值后 10 秒内生成补货事件，事件包含 warehouseId、shelfId、skuId 和 currentQuantity |

**关联 ADR**：ADR-005  
**优先级**：Should Have
