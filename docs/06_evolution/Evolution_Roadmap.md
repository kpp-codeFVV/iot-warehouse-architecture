# Evolution Roadmap - IoT 智能仓储监控与告警平台

本文件记录我们对后续架构演进的安排。

## 1. 当前架构版本（v1.0）现状与局限

v1.0 是当前验证版本。它通过 Docker Compose 启动 3 个 FastAPI 服务和 3 个中间件，覆盖设备接入、设备影子、库存事件、异常告警、边缘缓存补传和本地压测。

| 维度 | v1.0 已具备能力 | 当前局限 | 关联证据 |
| --- | --- | --- | --- |
| 设备接入 | `device-gateway` 接收模拟遥测消息，支持健康检查和边缘缓存 | 未接入真实硬件，认证策略简化 | ADR-001, ADR-002, PoC_Report |
| 设备状态 | `inventory-service` 维护设备最后状态和库存事件 | 设备影子未完整实现 desired/reported 版本确认 | ADR-004, DEBT-006 |
| 告警链路 | `alert-service` 生成 HIGH_TEMPERATURE 告警 | 告警规则配置、灰度发布和回滚能力有限 | ADR-005, DEBT-008 |
| 数据存储 | TimescaleDB/PostgreSQL 存储设备消息、设备影子、库存、事件和告警 | 单节点数据库不能代表生产容量，仍需补充 hypertable、压缩、保留和备份恢复策略 | ADR-003, DEBT-005 |
| 可靠性 | 边缘缓存与恢复补传 PoC 可验证 | 缓存容量水位、过期清理、补传限速未生产化 | ADR-006, DEBT-004 |
| 部署 | 本地一键启动，服务边界清晰 | 单机单实例，不代表高可用生产部署 | DEBT-001, DEBT-002 |
| 安全 | ADR 中设计设备认证、Topic ACL、TLS/mTLS | 原型未实现生产级证书体系 | QAS-006, DEBT-003 |

## 2. 演进触发条件

后续应看业务规模、质量指标或技术风险是否已经触发。

| 触发ID | 触发条件 | 说明 | 影响的质量场景 | 优先级 |
| --- | --- | --- | --- | --- |
| TR-001 | 设备规模超过 1,000 台，或单仓库设备类型超过 5 类 | 需要多仓库边缘节点注册、Topic 管理和配置下发 | QAS-002, QAS-008 | H |
| TR-002 | 告警从感知到生成多次超过 5 秒 | 当前事件通道或告警服务需要扩容、限流或异步解耦 | QAS-001, QAS-007 | H |
| TR-003 | 边缘缓存积压超过本地容量 70% | 需要缓存水位、清理策略、补传限速和积压告警 | QAS-003 | H |
| TR-004 | 需要接入真实设备或外部供应商设备 | 必须引入设备注册、证书发放、Topic ACL 和协议兼容策略 | QAS-006 | H |
| TR-005 | 查询历史数据窗口超过 24 小时或写入量持续增长 | 需要 TimescaleDB hypertable、压缩、保留策略和备份恢复 | QAS-005 | M |
| TR-006 | 告警阈值需要按仓库、货品、班次动态调整 | 需要规则服务、规则版本、灰度发布和回滚 | QAS-007 | M |
| TR-007 | 需要批量 OTA 升级真实设备 | 需要 OTA 任务服务、批次控制、失败重试和状态回传 | QAS-008 | M |
| TR-008 | 系统需要 7x24 小时运行或跨机房容灾 | 单机 Compose 必须演进为多副本、多可用区部署 | QAS-002, QAS-003 | H |

## 3. 演进路线图

### v1.0 -> v1.1（短期，0-6 个月）

**目标：** 把当前版本补强为“小规模试点版本”，优先处理会影响可靠性、配置修改和多仓库接入的债务。

| 关键变更 | 说明 | 偿还债务 | 关联ADR/QAS |
| --- | --- | --- | --- |
| 多仓库 Edge Registry | 记录 edge 节点、仓库、Topic 前缀、服务健康状态 | DEBT-010 | ADR-002, QAS-002 |
| Edge Cache 水位控制 | 增加缓存容量上限、过期清理、补传限速、积压告警 | DEBT-004 | ADR-006, QAS-003 |
| 告警规则版本化 | 将阈值从固定配置演进为规则表，支持启停、版本和回滚 | DEBT-008 | ADR-005, QAS-007 |
| 设备影子版本号 | 增加 desired/reported 结构、状态版本、最后确认时间 | DEBT-006 | ADR-004, QAS-004, QAS-008 |
| Docker 镜像固化 | 将运行时 `pip install` 改为构建阶段安装依赖 | DEBT-001 | 可运行原型 |
| 数据库治理 | 为 TimescaleDB 增加 hypertable、索引、保留策略、压缩和备份恢复演练 | DEBT-005 | ADR-003, QAS-002, QAS-005 |
| 基础观测指标 | 增加消息接入量、告警延迟、缓存积压、错误率指标 | DEBT-005 | QAS-001, QAS-002, QAS-005 |

**演进策略：**

1. 保持现有 REST API 兼容，新增字段使用可选字段。
2. 事件 Schema 增加 `schemaVersion`，消费者先兼容 v1 再逐步切换 v1.1。
3. 告警规则先以默认版本运行，再按仓库灰度启用。
4. Edge Cache 新策略先只记录水位告警，再启用清理和补传限速。

### v1.1 -> v2.0（中期，6-18 个月）

**目标：** 把试点版本演进为生产候选架构，重点解决高可用、安全、真实吞吐和数据生命周期。

| 关键架构变更 | 说明 | 偿还债务 | 关联ADR/QAS |
| --- | --- | --- | --- |
| 生产级 MQTT Broker | 从单节点 Mosquitto 演进到 EMQX、HiveMQ 或托管 MQTT 服务 | DEBT-002 | ADR-001, QAS-002 |
| 生产消息通道 | 根据吞吐和可靠性目标评估 Kafka、RabbitMQ 或云消息服务，降低同步 HTTP 转发压力 | DEBT-007 | ADR-005, QAS-001, QAS-009 |
| mTLS 与证书生命周期 | 建立设备证书签发、吊销、轮换、Topic ACL | DEBT-003 | ADR-001, QAS-006 |
| TimescaleDB 数据治理 | 引入 hypertable、压缩、保留策略、备份恢复演练 | DEBT-005 | ADR-003, QAS-005 |
| Kubernetes/云容器部署 | 服务多副本、滚动升级、健康检查和自动恢复 | DEBT-001 | ADR-002, QAS-002, QAS-003 |
| OTA 任务服务 | 支持设备升级批次、失败重试、暂停、回滚和状态回传 | DEBT-009 | ADR-004, QAS-008 |

**迁移策略：**

1. 先将事件消费者改为抽象接口，再迁移底层消息中间件。
2. MQTT Broker 采用双写或桥接方式迁移，避免设备一次性切换。
3. 安全策略采用“先观测、后阻断”：先记录证书和 ACL 违规，再逐步强制拒绝。
4. 数据库治理策略先在本地验证环境演练，再补充历史查询、保留策略和备份恢复校验。

### v2.0 -> v3.0（长期愿景）

**目标：** 面向跨区域、多仓库、大规模自动化仓储网络，支持更强的数据分析、数字孪生和自动运维能力。

| 方向 | 说明 | 主要收益 | 关联质量属性 |
| --- | --- | --- | --- |
| 多区域容灾 | 核心服务跨 Region 部署，关键数据具备容灾恢复策略 | 提升可用性和灾备能力 | 可靠性、可用性 |
| 数字孪生仓库 | 建立仓库、货架、设备、AGV 的实时状态模型 | 支撑可视化、仿真和调度优化 | 可观测性、可维护性 |
| 预测性补货 | 结合历史库存、销量和传感器数据预测补货需求 | 从规则触发演进到预测优化 | 业务及时性 |
| 智能告警降噪 | 使用规则、统计和机器学习降低重复告警 | 降低运维负担 | 可维护性、可观测性 |
| 自动化运维 | 自动扩缩容、自动故障隔离、自动回滚 | 支撑大规模持续运行 | 可用性、可伸缩性 |



## 4. 演进中的架构原则

| 架构不变量 | 说明 | 不能破坏的原因 |
| --- | --- | --- |
| 边云协同边界清晰 | Edge 负责接入、协议适配、缓存和本地恢复；Cloud 负责治理、存储、告警、分析和 API | 支撑 QAS-003，避免云端短暂不可用导致设备数据完全丢失 |
| 设备消息必须可追溯 | 每条遥测消息、异常事件、告警记录都应能通过 ID 和时间戳关联 | 支撑故障定位、审计和质量属性验证 |
| 事件契约优先 | 服务间异步集成必须先定义 JSON Schema 或等价契约 | 避免服务独立演进时破坏告警和补货链路 |
| 设备影子是设备状态来源 | 查询设备最后状态时优先读取设备影子，而不是临时日志 | 支撑 QAS-004 和 OTA 状态管理 |
| 演进必须保留 ADR 追溯 | 重大技术替换要新增 ADR 或将原 ADR 标记为 Superseded | 方便后续解释为什么改变原方案 |
| 原型结论必须声明边界 | 本地压测和 Docker Compose 只能说明链路表现，不能直接代表生产吞吐 | 避免把本地验证写成生产承诺 |
